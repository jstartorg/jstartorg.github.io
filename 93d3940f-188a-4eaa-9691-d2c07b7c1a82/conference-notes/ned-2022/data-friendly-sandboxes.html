Data-Friendly-Sandboxes

We’ve probably all heard it: Building & testing in sandboxes is a best practice and we should never build in production. But is it realistic? When we try to spin up a sandbox we can end up with no data at all, data that doesn’t match what we need or

The challenge: Sandboxes are only as good as the data that is in them

  • No data
  • Irrelevant data
  • Incomplete data

Do I reaaaally need to use sandboxes?

  • Yes. Use sandboxes to test, develop, demo and train features, onboard new staff members, keep up with new releases, backup metadata, and run QAT
  • Appendix slide has horror stories of not using them

Test data in production (don’t do it)

  • Dirty data
  • Negative impact on reporting
  • Confusing for end users
  • Building workarounds to deal with it
  • Insufficient for robust testing or training
  • Sets a bad example for other users

The rest of this isn’t relevant to us because we have OwnBackup, but I took notes anyway

Generating sandbox data

  • Partial sandboxes
    • Quick and easy to set up
    • 5GB of storage
    • Create reusable templates
    • Cheaper than full copy but not free
    • Can’t choose which records to copy
    • May not be able to copy all records
    • Limited number
    • Data isn’t anonymous
  • Create records manually
    • Create exactly what you need
    • No tools or training required
    • No additional cost
    • Time consuming
  • Create spreadsheets of data
    • Benefits of the above but better with larger data sets
    • Cumbersome with a lot of related records
  • Use forms or flows to generate records
    • Create multiple parent and child records
    • Choose to create the same data each time or take input for different data sets
    • Not feasible for large datasets
    • Link in presentation to a walkthrough
  • Third party tools
    • Mockaroo
      • Can create fake but realistic NPSP data for you
      • Free version has a limit of 1000 records per file
    • Snowfakery
      • Open source tool that generates fake data with relationships in one upload operation
      • Ideal for large data volumes and reuse of similar datasets
      • Open source community is collaborating on recipes
      • Uses YAML
      • Also good for stress testing for bulk updates

Importing sandbox data

  • Multistage process
    • Import parent records
    • Export newly created records with ids
    • Update spreadsheet with child record data including reference to parent record id
  • Keep repository of import files that will be used every time you refresh your sandbox (if you’re re-importing and want to keep the same parent child relationships, keep a field for old parent id on child records so you can map to the same one after it gets a new id)
  • NPSP Data Importer
    • Can upload data from multiple objects at once (standard/NPSP objects and any objects with relationships to those objects)
  • Salesforce tools
