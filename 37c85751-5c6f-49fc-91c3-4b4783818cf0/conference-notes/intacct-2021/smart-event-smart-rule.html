Smart Rule

  • basically SF validation rule but in reverse - if true continue, if false stop
  • fires before record event is committed
  • Event - really trigger event - on add, set(update), delete
  • works for most standard objects
  • you can have multiple per object, it really comes down to what the error message should be
  • runs them one after the other, throwing an error on the first failure but not evaluating the following ones, so you may have to keep clicking and clicking
    • so combine as many as you can up front most of the time
  • Cannot have conditionals in the error message, but can have regular merge fields

Smart Event - really smart action

  • Smart Events are like a single condition to action step in a Salesforce Flow
  • happens POST COMMIT ONLY
  • doesn't support all standard objects
  • Smart Events cannot loop over child items or lists of any kind
    • if it is an email, you can attach the PDF which does include line items etc - but you couldn't do it in the email

Conditions - a formula that returns true

  • condition is not required for a smart event, can be left empty

Event - really trigger event

  • triggered by user actions, API calls, or CSV import

Actions:

  • Email
  • Post to webhook
  • API call on Inacct instance
    • API Calls are INTACCT API calls, not just any old API
      • ALWAYS select Synchronous to avoid multiple smart events overwriting each other
      • NO SPACES ANYWHERE
      • SOAP/XML only, need to look at JSON to XML converters
  • write custom message to audit log (beyond the normal log messages)
    • Audit Log - there is already a system audit log, but if you want custom messages you can do this
